Для того чтобы подключение к вашей удалённой машине на Yandex Cloud происходило по HTTPS, 
вам нужно изменить подход в вашем deploy-job. 
Сейчас вы используете локальный Docker сокет (unix:///var/run/docker.sock), 
что подразумевает выполнение команд на той же машине, где запущен раннер. 
Для удаленного управления по HTTPS есть два основных пути: 
настроить Docker с удаленным доступом по TLS или использовать SSH-туннель.

Ниже описаны оба способа и дополнительные шаги для обеспечения безопасности (HTTPS для самого приложения).

Вариант 1: Настройка удаленного Docker daemon с TLS (рекомендуемый для Docker-инфраструктуры)
Этот способ позволяет вашим командам docker-compose выполняться на удаленном сервере так, 
как будто вы работаете на нем локально, но с полным шифрованием трафика.

Суть: Docker daemon на вашем сервере в Yandex Cloud слушает не только локальный сокет, но и сетевой порт (обычно 2376), 
используя TLS-сертификаты для шифрования и аутентификации.

Шаг 1: Настройка сервера (Ubuntu)
Подключитесь к серверу по SSH.

Создайте сертификаты. Обычно Docker генерирует их автоматически при включении удаленного доступа, 
но безопаснее сгенерировать свои. Вам понадобятся ключи CA, серверные ключи и клиентские ключи.

Настройте Docker service. Отредактируйте файл конфигурации Docker (например, /etc/docker/daemon.json), 
чтобы включить TLS и указать путь к сертификатам. Убедитесь, что в фаерволе открыт порт 2376.

Шаг 2: Настройка CI/CD
Вам нужно передать клиентские сертификаты в ваш deploy-job. Никогда не храните их в репозитории. 
Используйте переменные CI/CD типа "File" в GitLab.

В GitLab: Settings -> CI/CD -> Variables.

Добавьте три переменные типа "File":

DOCKER_CLIENT_CERT: содержимое файла cert.pem.

DOCKER_CLIENT_KEY: содержимое файла key.pem.

DOCKER_CA_CERT: содержимое файла ca.pem.

Также добавьте переменную DOCKER_HOST_REMOTE со значением tcp://<ваш-публичный-IP-адрес>:2376.

Шаг 3: Модификация gitlab-ci.yml
В вашем deploy-job нужно скопировать эти "файлы" из переменных в контейнер и настроить окружение Docker.

yaml
deploy-job:
  stage: deploy
  image: docker:27-cli  # Оставляем ваш образ
  before_script:
    - echo "==================== start deploy-job ===================="
    # Устанавливаем переменные для удаленного подключения
    - export DOCKER_HOST=$DOCKER_HOST_REMOTE
    - export DOCKER_TLS_VERIFY=1
    # Создаем директорию для сертификатов
    - mkdir -p $HOME/.docker
    # Копируем сертификаты из переменных GitLab (они приходят как файлы)
    - cp $DOCKER_CA_CERT $HOME/.docker/ca.pem
    - cp $DOCKER_CLIENT_CERT $HOME/.docker/cert.pem
    - cp $DOCKER_CLIENT_KEY $HOME/.docker/key.pem
    - chmod 600 $HOME/.docker/key.pem
    # Проверяем подключение
    - docker info
  script:
    - docker compose build
    - docker compose up -d
  after_script:
    - echo "==================== finish deploy-job ===================="
  dependencies:
    - build-job
    - package-job
Этот метод описан в обсуждениях на Stack Overflow как надежный способ удаленного управления .

Вариант 2: Использование SSH (более простой и универсальный)
Этот метод имитирует ручной вход на сервер. Вы подключаетесь по SSH и выполняете команды внутри удаленной сессии.

Шаг 1: Настройка сервера и SSH-ключей
Сгенерируйте пару SSH-ключей (без парольной фразы) специально для GitLab CI/CD.

Публичный ключ добавьте на вашем сервере Yandex Cloud в файл ~/.ssh/authorized_keys пользователя, 
под которым будете деплоить (например, ubuntu или gitlab-runner).

Приватный ключ сохраните.

Шаг 2: Настройка CI/CD
В GitLab: Settings -> CI/CD -> Variables.

Добавьте переменную типа "File" с именем SSH_PRIVATE_KEY и вставьте содержимое вашего приватного ключа.

Добавьте переменную SSH_KNOWN_HOSTS. Чтобы получить её значение, выполните локально команду: 
ssh-keyscan -H <IP-адрес-вашего-сервера>. Скопируйте вывод и вставьте в переменную типа "File" .

Шаг 3: Модификация gitlab-ci.yml
Вам нужно будет установить SSH клиент, запустить ssh-agent и выполнять команды через ssh.

yaml
deploy-job:
  stage: deploy
  image: alpine:latest  # Легковесный образ, в котором мы установим SSH и Docker клиент
  before_script:
    - echo "==================== start deploy-job ===================="
    # Устанавливаем openssh-client и docker клиент (для compose, если нужен)
    - apk add --no-cache openssh-client docker-cli docker-cli-compose
    # Запускаем ssh-agent и добавляем ключ
    - eval $(ssh-agent -s)
    - chmod 600 "$SSH_PRIVATE_KEY"
    - ssh-add "$SSI_PRIVATE_KEY"
    # Создаем known_hosts
    - mkdir -p ~/.ssh
    - chmod 700 ~/.ssh
    - cp "$SSH_KNOWN_HOSTS" ~/.ssh/known_hosts
    - chmod 644 ~/.ssh/known_hosts
  script:
    # Копируем файлы на сервер (например, docker-compose.yml и .env файлы)
    - scp docker-compose.yml user@your-server-ip:/path/to/app/
    - scp .env user@your-server-ip:/path/to/app/
    # Выполняем команды на удаленном сервере
    - ssh user@your-server-ip "cd /path/to/app && docker compose pull && docker compose up -d"
  after_script:
    - echo "==================== finish deploy-job ===================="
  dependencies:
    - build-job
    - package-job
Этот подход с SSH детально описан в официальной документации GitLab .

Как настроить HTTPS для самого приложения (Nginx + Let's Encrypt)
В вашем вопросе также идет речь о подключении к машине через HTTPS. 
Переключение CI/CD на HTTPS не делает автоматически ваше приложение доступным по HTTPS. 
Для этого нужно настроить веб-сервер на самой машине.

Изучив примеры успешных проектов, можно предложить стандартную связку Nginx + Let's Encrypt :

Nginx как reverse-proxy: Установите Nginx на вашем сервере. 
Он будет слушать порты 80 (HTTP) и 443 (HTTPS) и перенаправлять трафик к вашим Docker-контейнерам.

Certbot для получения сертификатов: 
Используйте Certbot от Let's Encrypt для автоматического получения и обновления SSL-сертификатов.

Автоматизация: Вы можете автоматизировать первоначальное получение сертификатов через CI/CD, 
передавая домен и email как переменные . 
Например, ваш deploy-job после деплоя контейнеров может выполнить скрипт на сервере для запуска Certbot.

Важный момент про доступ к реестру: 
Поскольку ваш CI/CD пайплайн (например, в package-job) загружает образы, 
а раннеру на сервере нужно их скачать, убедитесь, 
что ваш GitLab Runner на сервере может подключиться к вашему Container Registry. 
Для этого также может потребоваться настроить переменные с DOCKER_AUTH_CONFIG для аутентификации в приватном реестре .

Таким образом, ваш путь сейчас — 
выбрать между TLS-сертификатами для Docker (Вариант 1) и SSH-туннелем (Вариант 2) 
для безопасного управления сервером из CI/CD. 
А затем, отдельно, настроить Nginx с Let's Encrypt для безопасного доступа пользователей к вашему приложению.
