Вот пошаговый план действий, который разделен на настройку сервера и интеграцию с вашим CI/CD.

                                        Часть 1: Настройка сервера (выполняется один раз по SSH)

Эти шаги нужно выполнить вручную или через скрипт на вашей удаленной машине.

                                                    Подготовка сервера и DNS

Убедитесь, что у вас есть доменное имя, которое указывает на публичный IP-адрес вашего сервера в Yandex Cloud (A-запись). 
HTTPS не работает с IP-адресом, только с доменом .

Подключитесь к серверу по SSH.

Откройте порты 80 (HTTP) и 443 (HTTPS) в фаерволе вашей ОС (например, ufw) и 
обязательно в сетевых настройках (security group) вашего инстанса в Yandex Cloud .

                                                    Установка Nginx и Certbot

Обновите список пакетов и установите Nginx, Certbot и плагин для Nginx :

bash
sudo apt update
sudo apt install -y nginx certbot python3-certbot-nginx
Настройка Nginx как Reverse Proxy

Создайте конфигурационный файл для вашего сайта. Замените yourdomain.com на ваш реальный домен :

bash
sudo nano /etc/nginx/sites-available/yourdomain.com
Вставьте следующую базовую конфигурацию. Она говорит Nginx слушать порт 80 и перенаправлять все запросы на тот порт, 
где работает ваше приложение внутри сервера. Предположим, ваше приложение доступно на localhost:8080. 
Если порт другой, исправьте proxy_pass:

nginx
server {
    listen 80;
    server_name yourdomain.com www.yourdomain.com;

    location / {
        proxy_pass http://localhost:8080; # Адрес вашего Java-приложения
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;
    }
}
Сохраните файл и активируйте его, создав символическую ссылку :

bash
sudo ln -s /etc/nginx/sites-available/yourdomain.com /etc/nginx/sites-enabled/
sudo nginx -t # Проверка конфигурации
sudo systemctl reload nginx # Применение конфигурации
Теперь ваше приложение должно быть доступно по HTTP через ваш домен.

                                            Получение SSL-сертификата с помощью Certbot

Запустите Certbot с плагином для Nginx. Он автоматически получит сертификат и изменит конфигурацию Nginx для использования HTTPS :

bash
sudo certbot --nginx -d yourdomain.com -d www.yourdomain.com
Certbot задаст несколько вопросов (email, согласие с условиями). Следуйте инструкциям. 
В конце он спросит, нужно ли перенаправлять весь HTTP-трафик на HTTPS. Выберите "2: Redirect" .

Готово! Certbot обновит ваш конфиг Nginx, добавив в него настройки для SSL и перенаправления с HTTP на HTTPS.

Автоматическое обновление сертификата

Сертификаты Let's Encrypt действуют 90 дней. Certbot автоматически устанавливает системный таймер для их обновления. 
Проверьте, что он активен :

bash
sudo systemctl status certbot.timer
sudo certbot renew --dry-run # Тест процесса обновления


                                                    Часть 2: Интеграция с вашим GitLab CI/CD

Теперь, когда сервер настроен, ваш пайплайн (deploy-job) должен не просто запускать контейнеры, 
но и гарантировать, что конфигурация Nginx остается актуальной (хотя в данном случае это необязательно, если она не меняется). 
Главное — убедиться, что ваше Java-приложение слушает запросы на том порту, который указан в proxy_pass (в примере выше это 8080).

Вы можете оставить ваш docker-compose.yml без изменений, просто убедившись, 
что порт приложения (например, 8080) проброшен на хост-машину, 
но не открыт наружу (то есть слушает 127.0.0.1:8080 или просто 8080, 
но файрвол на сервере и в облаке не должен открывать этот порт для внешнего мира). 
Вся внешняя коммуникация будет идти через Nginx на порты 80/443.

                                                        Автоматизация через deploy-job:

Вы можете добавить шаги в script вашего deploy-job, 
чтобы после деплоя нового кода обновлять конфигурацию Nginx или даже получать сертификат, 
если он еще не был получен (например, при первом деплое). 

Однако самый простой и надежный способ — настроить сервер один раз вручную по инструкции выше, 
а ваш deploy-job оставить как есть, отвечающим только за сборку и запуск Docker-контейнеров с новой версией приложения.

В вашем deploy-job уже есть подключение по SSH 
(через DOCKER_HOST: unix:///var/run/docker.sock означает, что вы работаете с локальным демоном Docker на сервере). 
Для применения изменений Nginx через CI/CD вам нужно будет выполнять команды на сервере через SSH, но это уже усложнение.

                                                                Итоговая архитектура

После выполнения этих шагов трафик будет идти по следующему пути:

Браузер пользователя (HTTPS) --> 
Ваш сервер (порт 443) --> 
Nginx (расшифровывает HTTPS) --> 
http://localhost:8080 --> 
Ваше Java-приложение в Docker

Таким образом, вы получаете полноценное HTTPS-соединение с доверенным сертификатом, 
используя существующую инфраструктуру Docker и автоматическое обновление сертификатов
